# Architecture Context — Fincept AI Ops

## Stack

| Layer | Technology | Role |
|---|---|---|
| API Framework | FastAPI (Python 3.11+) | HTTP entrypoint, route handlers, middleware |
| Vercel Adapter | `api/index.py` (ASGI handler) | Serverless deployment adapter |
| MCP Server | `mcp/server.py` (StdIO) | Tool exposure layer for AI agent integration |
| Connectors | `apps/fincept_aiops/connectors/` | Data adapters for all external sources |
| Agents | `apps/fincept_aiops/agents/` | Risk evaluation and supervised execution logic |
| N8N Workflows | `workflows/*.json` | Orchestration of research and briefing pipelines |
| Infra | Docker + docker-compose | Local and self-hosted deployment |
| CI/CD | GitHub Actions (`ci.yml`) | Test matrix (py3.11 + py3.12) + Docker build |

## System Boundaries

- `api/` — Vercel-compatible ASGI entrypoint only. No business logic lives here.
- `apps/fincept_aiops/` — All application logic: connectors, agents, middleware, risk policy, app factory.
- `mcp/` — MCP StdIO server. Exposes 7 tools for AI agent consumption. No side effects beyond read + paper execution.
- `workflows/` — N8N workflow JSON exports. Orchestrates the research pipeline, briefing generation, and approval triggers.
- `tests/` — All pytest test cases. 17 cases across 4 modules. No test modifies production state.
- `data/` — Runtime state: paper portfolio, signal history, audit log. Never committed with real values.
- `docs/` — Architecture decisions, API contracts, deployment guide, ops playbook, context files.

## Storage Model

- **`data/portfolio.json`** — Paper positions and balance. Written only by `execution_ops` agent after dual-gate approval.
- **`data/audit.log`** — Append-only audit trail. Every execution attempt — approved or rejected — is logged here.
- **`data/signals/`** — Signal candidates generated by the research pipeline. Read by `risk_guard` for evaluation.
- **No database.** MVP uses file-based state. PostgreSQL is a Phase 4+ decision, not current scope.
- **No blob storage.** All artifacts are local files. No S3 or external storage in MVP.

## Auth and Access Model

- MVP has no multi-user auth. The system is single-operator: Mehmet Zeki.
- API endpoints are protected by `API_KEY` header validated in `middleware.py`.
- Human approval gate is manual: operator must set `HUMAN_APPROVAL=true` in environment before execution.
- No OAuth, no Clerk, no JWT in MVP. Auth complexity is deferred to Phase 3+.

## Invariants

1. **No live broker calls.** All execution routes to `broker_sandbox` only. The live broker connector is not instantiated in MVP.
2. **No execution without human approval.** `execution_ops` agent will not call `broker_sandbox.place_order()` unless `HUMAN_APPROVAL=true` env var is explicitly set.
3. **No risk override via prompt.** `risk_guard` output is never overridden by agent instruction. Risk policy is code-only (`risk_policy.py`).
4. **No long-lived background work in request handlers.** FastAPI handlers return fast. All pipeline work runs in N8N or scheduled jobs, not inside HTTP requests.
5. **Max 5 active connectors.** Exactly: `market_data`, `fundamentals`, `news`, `backtest`, `broker_sandbox`. Any additional connector requires an architecture decision entry.
6. **Max 6 agents.** Current: `risk_guard`, `execution_ops`. Headroom exists. No new agent without a documented role in `AGENTS.md`.
